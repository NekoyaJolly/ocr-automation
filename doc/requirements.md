# OCR Automation App — 要件定義書 (v2)

> **v2 改訂の背景**
> v1 では NDL OCR Lite を使ったローカル完結 OCR を前提としていたが、手書き日本語の精度が業務要件を満たさないことが Phase 2 完了時の精度検証で判明した。v2 では Google Gemini 3.1 Pro Preview を使ったクラウド OCR + 構造化抽出方式に切り替え、デスクトップアプリ ↔ 自前バックエンド ↔ Gemini API の3層構成とする。

## 1. プロジェクト概要

### 1.1 目的
スキャン画像・撮影画像から手書き文字を OCR で読み取り、Gemini による構造化抽出を行い、ユーザーが定義したテンプレートに沿った複数フォーマットで自動的に出力し、さらに自動印刷までを一気通貫で実行するデスクトップアプリケーション。

### 1.2 コアバリュー (North Star)
**「入り口と出口だけが人間、間は全部自動」**

- 入り口: スキャナで紙を取り込む or ユーザーがフォルダに画像をドラッグ
- 間: ファイル検知 → 画像送信 → OCR + 構造化抽出 → テンプレート適用 → ファイル生成 → 自動印刷 (全て自動)
- 出口: プリンタから出てきた紙を受け取る、または出力ファイルを配布する

このコアバリューは全ての設計判断における最優先指針とする。

### 1.3 対象ユーザー
- 主に日本語(漢字・かな・英数字混在)の手書き文書を扱う企業
- 受付・バックヤード等で IT リテラシーの高くないスタッフが運用する想定
- 配布対象規模: 最大 30 社程度
- 業務例: 手書き納品書から請求書・領収書・控えを自動生成して即印刷

### 1.4 動作環境
- Windows 10 以降 (64bit)
- macOS 12 (Monterey) 以降
- インターネット接続必須(バックエンド経由で Gemini API を使用)
- GPU 不要

## 2. 機能要件

### 2.1 コア機能

#### F-01: 入力フォルダ監視(サブフォルダベース運用)
- ユーザーが指定した「入力ルートフォルダ」配下のサブフォルダを再帰的に監視
- 各サブフォルダはテンプレートセットと 1:1 で紐付けられる
  - 例: `入力/納品書/` → 納品書セット、`入力/領収書/` → 領収書セット
- サブフォルダとテンプレートセットの対応は GUI で設定し、設定ファイルに永続化
- 新規ファイル(対応画像形式: jpg, jpeg, png, tiff, tif, jp2, bmp, pdf)が追加されたことを検知
- ファイル書き込み完了を待ってからトリガー
- 監視は GUI 起動中バックグラウンドで継続

#### F-02: 画像送信 + OCR + 構造化抽出
- 検知した画像ファイルをバックエンド API へ HTTPS で送信
- リクエストには画像 + 適用するテンプレートの抽出プロンプト + JSON Schema を含める
- バックエンドが Gemini API を呼び出し、構造化された JSON を返す
- レスポンスをローカルで受け取り、後続のテンプレート適用処理に渡す
- ネットワークエラー時は失敗扱い(リトライは F-04b 参照)

#### F-03: テンプレートセット適用
- 1 つの OCR 結果に対して、複数のテンプレートを同時に適用可能
- テンプレートセット = 複数のテンプレート + 各テンプレートの有効/無効 + 各テンプレート個別の出力先・印刷設定
- 各テンプレートは独立した「抽出プロンプト + JSON Schema + 出力配置」を持つ
- 出力フォーマットはテンプレートごとに選択可能: txt / docx / xlsx / pdf
- **部分成功を許容**: セット内の一部テンプレート出力が失敗しても、成功した分は出力する
- 単一テンプレートのみのセット(従来の 1:1 ケース)も同じ仕組みで扱える

#### F-04: 出力フォルダ書き出し
- テンプレート適用後のファイルをユーザー指定の出力フォルダへ保存
- セット内の各テンプレートは個別のサブフォルダへ出力可能(例: `出力/invoices/`, `出力/receipts/`)
- ファイル名は命名規則(タイムスタンプ・元ファイル名・抽出フィールド値などの組み合わせ)に従う

#### F-04b: 失敗時の自動リトライ
- セット内のテンプレート適用または出力で失敗が発生した場合、失敗した**テンプレート単位**でリトライ
- リトライ上限: **2 回**(初回 + リトライ 2 回 = 最大 3 回試行)
- リトライ間隔: 指数バックオフ(1秒 → 3秒)
- ネットワークエラー(バックエンド通信失敗)も同じリトライ機構の対象
- 全リトライ失敗後、対象ファイルと失敗情報を「失敗フォルダ」へ移動
- 部分成功(一部テンプレートのみ失敗)の場合、成功分は通常出力フォルダ・失敗分のみ失敗フォルダへ記録

#### F-05: 自動印刷(テンプレート単位制御)
- 出力フォルダに新規ファイルが追加されたことを検知し、自動的にプリンタへ送信
- 印刷の有効/無効は**テンプレート単位**で制御可能
- テンプレートセット定義内に各テンプレートの `auto_print` フラグを持つ
- 印刷先プリンタはユーザーが事前に選択(テンプレート単位で別プリンタも指定可能)
- 印刷成功/失敗のログを記録

#### F-06: ライセンスキー認証
- 初回起動時にユーザーがライセンスキーを入力
- ライセンスキーはバックエンド (Firestore) で検証
- 検証成功後、キーを暗号化してローカル設定に保存
- 以降の API 呼び出しは全てライセンスキーをヘッダに付与
- 無効・期限切れ・取り消されたキーの場合はエラー表示し機能停止

### 2.2 GUI 機能

#### F-10: メインウィンドウ
- 監視状態(動作中/停止中)を視覚的に表示
- リアルタイム処理ログを表示(直近 100 件程度をスクロール表示)
- 開始/停止ボタン
- 設定タブ・ログタブのタブ構成
- ライセンス状態の表示(残り日数、利用回数など)

#### F-11: フォルダ設定画面
- 入力ルートフォルダ・出力ルートフォルダ・失敗フォルダの3つを GUI で選択
- 入力ルート配下のサブフォルダ一覧を表示し、各サブフォルダにテンプレートセットを割り当てる UI を提供
- フォルダパスの可視化と「フォルダを開く」ボタン
- 設定は永続化(次回起動時に復元)

#### F-12: テンプレートエディタ
- テンプレートの新規作成・編集・削除・複製
- **抽出プロンプトのエディタ**(Gemini への指示文)
- **JSON Schema エディタ**(レスポンスの構造を定義、Gemini structured output で強制)
- **出力配置エディタ**(抽出された値を出力フォーマットのどこに配置するか)
  - xlsx: セル番号指定
  - docx: ブックマーク or 段落番号指定
  - pdf: テンプレート PDF のフォームフィールド名
- 出力フォーマット(txt/docx/xlsx/pdf)の選択
- ファイル命名規則の定義(抽出されたフィールド値を変数として使用可能)
- テンプレートのプレビュー機能(サンプル画像で動作確認)

#### F-12b: テンプレートセット管理画面
- テンプレートセットの新規作成・編集・削除・複製
- セットに含めるテンプレートの追加/削除/有効化/無効化
- 各テンプレートの個別設定: 出力サブフォルダ・自動印刷 ON/OFF・使用プリンタ
- セットのプレビュー(セット適用時にどのファイルが何個生成されるかを事前確認)

#### F-13: プリンタ設定画面
- システムに登録されているプリンタの一覧表示
- デフォルトプリンタの選択
- 自動印刷の ON/OFF 切り替え
- 印刷部数・用紙サイズなどの基本オプション

#### F-14: ライセンス設定画面
- 現在のライセンスキー表示(マスク表示)
- ライセンス情報(契約会社名、有効期限、当月利用数、月間上限)
- ライセンスキーの再入力・変更
- バックエンドへの接続テストボタン

### 2.3 サブ機能

#### F-20: ログ管理
- 全ての処理(監視・送信・OCR・出力・印刷)をログファイルに記録
- ログレベル: INFO / WARNING / ERROR
- ログファイルのローテーション(日次)

#### F-21: 設定の永続化
- ユーザー設定は JSON 形式でローカル保存
- ライセンスキーは別途暗号化保存(OS のキーチェーン等を利用)
- 保存先はユーザーホーム配下の専用ディレクトリ
  - Windows: `%APPDATA%/OCRAutomation/`
  - macOS: `~/Library/Application Support/OCRAutomation/`

#### F-22: テンプレート/テンプレートセットのインポート/エクスポート
- テンプレートファイル(YAML)とテンプレートセットファイル(YAML)を別 PC に持ち運べるよう、エクスポート/インポート機能を提供
- セットをエクスポートする際は、参照されているテンプレートも一緒に zip にまとめる

## 3. 非機能要件

### 3.1 パフォーマンス
- 1枚の A4 画像(300dpi)に対する処理時間(検知 → API 送信 → 結果受信 → ファイル出力): 15秒以内が目標
- バックエンドへの画像送信は適切に圧縮(オリジナル画像の解像度を保ちつつ、不要なメタデータは除去)
- GUI の応答性: 処理中も UI がフリーズしないこと(別スレッドで実行)

### 3.2 信頼性
- ネットワーク一時障害には自動リトライで対応
- バックエンド長期障害時はエラー表示し、ファイルは失敗フォルダへ
- アプリ予期せぬクラッシュ時、再起動で監視状態を自動復元

### 3.3 セキュリティ
- 画像送信は HTTPS のみ(平文通信禁止)
- ライセンスキーは平文保存禁止(OS キーチェーン or 暗号化保存)
- バックエンドは Gemini API キーをアプリに渡さず、サーバー側で完結
- バックエンドへの通信は全てライセンスキー認証

### 3.4 配布性
- **完全バンドル方式**: Python ランタイム・全依存ライブラリを単一インストーラに内包
- ユーザー側での Python インストール・依存ライブラリ導入は一切不要
- インストーラ形式: Windows は `.exe`、macOS は `.app` (`.dmg`)
- 配布物のサイズは 100〜200MB を目標(v1 の 1〜2GB から大幅減)

### 3.5 拡張性
- OCR エンジンは抽象インターフェース経由で呼び出し、将来的に別エンジン(Cloud Vision、ローカルモデル等)へ差し替え可能な構造とする
- 出力フォーマットも抽象化し、新規フォーマット(例: HTML, Markdown, JSON)の追加を容易にする
- バックエンド API のバージョニングを行い、クライアント・サーバーの互換性を保つ

### 3.6 ライセンス
- アプリ本体ライセンス: 未定(MIT or Apache 2.0 想定)
- Gemini 利用規約に準拠
- ユーザーへの利用規約・プライバシーポリシー提示は配布前に整備

## 4. 技術スタック(確定)

### 4.1 デスクトップアプリ
- 言語: Python 3.12
- GUI: PySide6 (Qt6)
- フォルダ監視: watchdog
- HTTP クライアント: httpx (async 対応)
- データモデル: pydantic v2
- テンプレート定義: PyYAML
- Word 出力: python-docx
- Excel 出力: openpyxl
- PDF 出力: reportlab + pypdf (テンプレート PDF への書き込み)
- ログ: 標準 logging + RotatingFileHandler
- 印刷(Win): pywin32
- 印刷(Mac): subprocess + lp コマンド
- ライセンスキー暗号化: keyring (OS キーチェーン経由)

### 4.2 バックエンド
詳細は `docs/backend/backend-requirements.md` 参照
- 言語: Python 3.12
- フレームワーク: FastAPI
- インフラ: GCP Cloud Run
- データベース: Cloud Firestore
- LLM: Google Gemini 3.1 Pro Preview (マルチモーダル + structured output、モデル ID 例: `gemini-3.1-pro-preview`)

### 4.3 ビルド・配布
- パッケージング: PyInstaller(`--onedir` 形式)
- CI ビルド: GitHub Actions(Windows / macOS マトリクス)
- バージョン管理: Git + GitHub
- 依存管理: uv

## 5. 制約事項

### 5.1 ネットワーク前提
- 完全オフライン動作は **不可**(v1 からの方針変更)
- ユーザー環境に安定したインターネット接続があることが前提
- バックエンド側がダウンしている場合、アプリは機能停止

### 5.2 コスト構造
- Neko (開発者) 側でバックエンドと Gemini API のコストを負担
- Google AI Ultra 加入による $100/月 GCP クレジットを活用
- 30社規模での試算では、無料枠 + クレジット内で完結する見込み

### 5.3 機能スコープ外
- 完全オフライン動作
- クラウド連携(Dropbox, Google Drive 直接連携など)
- 複数 PC 間の設定同期
- ユーザー認証・マルチユーザー対応(1 PC = 1 ライセンス前提)
- OCR 結果の手動補正 UI(初版では対応しない)
